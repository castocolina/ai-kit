## E1 — review-spec skill + framework-aware fix routing

**Intent**: today `/review-spec` orchestrates `reviewing-specs` (reviewer) then
`applying-review-feedback` (own fixer). The fixer is wrong when the document was
authored by a framework that owns its own authoring/edit loop. The rewrite step
must be routed to the framework's native authoring tool.

- **FR-1.1** — **Decided: convert `/review-spec` from a command into an orchestrator
  skill** (`skills/review-spec/`). Rationale: the kit prizes Agent-Skills portability
  (works on OpenCode/Gemini/etc. that have no slash commands) and consistency with the
  skills-everything model (superpowers orchestrates via skills). Being an orchestrator
  does not require a command — a skill can dispatch subagents and invoke other skills,
  and can still be triggered explicitly. The current `commands/review-spec.md` migrates
  to a `SKILL.md`.
- **FR-1.2** — Detect the document's originating framework, reusing the framework
  classification `reviewing-specs` already performs (do not build a second detector).
- **FR-1.3** — **Routing of the rewrite step** by provenance:
  - **superpowers** (doc generated by `brainstorming` or `writing-plans`) → the
    rewrite MUST be done by the corresponding **`brainstorming`** / **`writing-plans`**
    skill, **not** `applying-review-feedback`.
  - **GSD** → edit via **`gsd-planner`** or the **`/gsd-plan-phase --reviews`** skill,
    then validate with the **`gsd-plan-checker`** agent — **not** the own fixer.
  - **generic / no framework** → keep the current `applying-review-feedback` fixer.
- **FR-1.4** — The reviewer (`reviewing-specs`) stays framework-aware and unchanged
  in role; only the **fix/rewrite** stage is re-routed.
- **FR-1.5** — `/review-spec` documents the routing table (which tool edits what)
  so the behavior is transparent.

**Execution tooling**: when building/refining the command and its skills, run
**`/skill-judge:skill-judge`** to evaluate their design quality against the spec.

**Dependencies**: none (independent of the status-line epics).

**Resolved**: FR-1.1 → skill (above). Ambiguous/low-confidence framework detection
→ fall back to the generic `applying-review-feedback` fixer, and the skill states
the detected framework so the routing is transparent.

## E2 — mermaid-audit color-palette guidance

**Intent**: extend `mermaid-audit` beyond syntax/layout to assess and guide the
**color palette** of diagrams. Mermaid defaults are too gray/flat and carry no
emphasis.

- **FR-2.1** — mermaid-audit reviews each diagram's palette, not just render success and layout.
- **FR-2.2** — Flag low-emphasis defaults (too gray/flat, poor figure-ground separation).
- **FR-2.3** — Recommend **sober, non-garish** palettes that encode semantics:
  layers, component types/levels, responsibility, and old-vs-new distinctions.
- **FR-2.4** — Provide **curated palettes** (Material-design-like; svelte/react-like)
  and concrete guidance on choosing and **combining** them (when to use accent vs neutral, etc.).
- **FR-2.5** — Emit actionable suggestions (e.g. the `classDef`/theme snippet to apply).
- **FR-2.6** — Advisory accessibility guidance: prefer color-blind-safe combinations
  and adequate contrast; surface a warning when a recommended pairing is risky.

**Decided**: **recommend + emit the ready-to-paste snippet only** (no auto-edit of
the diagram), consistent with mermaid-audit being an auditor. FR-2.6 accessibility
checks are **advisory** (warnings/suggestions), not a hard gate.

**Plan note**: implemented as a static `scripts/mermaid_style.py` analyzer with
near-binary C-rules (color), S-rules (node shape), and render-measured R-rules
(geometry / aspect ratio), plus a consensus eval that proves the rules are
reproducible across independent agents. Two axes added beyond the color-only FR-2.x
per user emphasis: **node-shape semantics** and **render geometry** ("forma" = a
diagram too horizontal — ratio < 0.4 — or scrolling past ~3 pages reads badly;
target vertical-leaning, in-band). Accessibility (C2/C5) stays advisory.

**Execution tooling**: build/extend this skill with **`/skill-creator:skill-creator`**
(and validate with `/skill-judge` if useful).

**Dependencies**: none.

## E3 — status-line bug fixes

**Intent**: correctness fixes and platform fallbacks in `tools/status-line.py`.
Consumes E4's palette/toggles where relevant.

- **FR-3.1** — Detect `effort=auto` (currently misreported as `high`). Investigate
  the real source (`raw["effort"]["level"]` / `CLAUDE_EFFORT`) and classify `auto` correctly.
- **FR-3.2** — Render `auto` distinctively: **each letter a different color**, and
  the effort "ladder" (`▁▃▄▆█`) colored to match.
- **FR-3.3** — Make the default **blue** robust where it reads as **purple** on some
  screens (pick a better default SGR; user override lands via E4 `[palette]`).
- **FR-3.4** — Fix the **memory** segment: in wezterm it shows `<10mb`, indicating the
  wrong process is measured. Correct process/RSS selection.
- **FR-3.5** — Add a **macOS fallback** for process RSS (memory).
- **FR-3.6** — Add a **macOS fallback** for terminal **columns/rows** detection.
- **FR-3.7** — Color the **chat-size** segment (`seg_chat_size`, the 💾 disk-shaped
  indicator), today **uncolored**, by **size thresholds** — mirroring how the context
  segment colors via `pick_color(value, ramp)`. Add a `CHAT_SIZE_RAMP` keyed on bytes:
  **< 5 MB → neutral/default**, **≥ 5 MB → red + bold**, **≥ 10 MB → magenta/purple**
  (the `RED`/`MAGENTA` SGRs are already bold). Thresholds follow the same "first ceil the
  value is below wins" ramp shape as `CONTEXT_RAMP`, and land as overridable values via E4's
  config like the other E3 colors.

**Execution tooling**: TDD (extend `tests/test_status_line.py`); no skill involved.

**Dependencies**: E4 (uses palette overrides + segment toggles); FR-3.1/3.2 also
align with E4's effort segment; FR-3.7's thresholds/colors are overridable via E4.

**Resolved**: `auto` (FR-3.2) renders with a fixed rotating cycle
`CYAN → GREEN → YELLOW → ORANGE → MAGENTA → BLUE` across its letters and ladder bars
(distinct from the static per-effort colors). The wezterm memory bug (FR-3.4) is an
**implementation investigation** — capture a wezterm sample; likely the wrong PID /
a parent process is being measured — tracked as an impl task, not a spec ambiguity.

## E4 — status-line config & extensibility  *(PRD drafted)*

See **`statusline-config-extensibility-v1.0-prd.md`**. Summary of locked decisions:
- Three-tier config: **internal defaults < `~/.config/ai-kit/statusline.toml` (TOML) < env (`CC_AI_KIT_*`)**.
- Segment toggles via `CC_AI_KIT_SEGMENT_<KEY>` (bool grammar) and `[segments]`.
- Layout overridable via `[[line]]` (all-or-nothing).
- External drop-in segments (`~/.config/ai-kit/segments/`) placed by header, TTL-cached.
- Minimal `[palette]` overrides.
- Editable surface (`SEGMENTS`/`LAYOUT`) moved to top of file.
- A complete, fully-commented **recipe** shipped + copied to the default path if absent.

**Execution tooling**: TDD. **Dependencies**: none; **feeds** E3 and E5.

## E5 — installer ergonomics + setup wizard

**Intent**: make install fast and configurable; the `.sh` becomes a thin
downloader, the real logic a Python wizard. Consumes E4's config + recipe.

- **FR-5.1** — Provide a **`Makefile`** with `make install` (and friends) for users
  who clone the repo — fast, with shell autocomplete.
- **FR-5.2** — `install.sh` **detects its mode**: invoked directly with a local repo
  present → use the local checkout; invoked on-the-fly (piped from `curl`) → download
  the repo first. It becomes a thin **wrapper/bootstrapper**.
- **FR-5.3** — A **Python wizard/setup** drives configuration: first asks whether the
  user wants the **skills**, the **status line**, or both.
- **FR-5.4** — The wizard offers **per-option toggles with a clear visual state**:
  `[x]/[ ]` (or equivalent) selected/unselected markers, the option name in an
  **accent color when enabled and dimmed when disabled**, a one-line note of **what
  each option delivers**, and a **live preview line** of the resulting status line.
- **FR-5.5** — **Double-confirm** before changing `statusLine` if a *different* status
  line is already configured in `settings.json`.
- **FR-5.6** — A **reconfigure** entry point to re-run the wizard later.
- **FR-5.7** — The wizard writes/regenerates `~/.config/ai-kit/statusline.toml` from
  the E4 recipe (with internal defaults as the floor).

**Execution tooling**: TDD for the Python wizard; bash/shellcheck for the wrapper + Makefile.

**Dependencies**: **E4** (config model + recipe).

**Decided**: wizard UX is **plain stdin prompts with zero heavy TUI dependencies**,
but with ANSI affordances (markers, accent/dim, per-option example) per FR-5.4. The
live preview renders by feeding a **representative sample JSON to `status-line.py`**
and printing its output.
